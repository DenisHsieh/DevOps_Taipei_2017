2017 年的軟體

  • 趨勢變化
  • 競爭激烈
  • 時間就是金錢

如何改變？

適應改變，改變是很劇烈的 -> 敏捷開發
改變是否符合市場的需求

  • 提早確認
  • 修正方向

產品團隊的挑戰

  • 持續交付新產品
  • 保持系統穩定
  • 了解服務狀態
  • 應付意外狀況

持續交付高品質軟體

軟體開發來說，最困難的不是如何上線…
是如何快速、持續把新版本上線…

持續交付的痛點

  • 穩定性
  • 相容性
  • 可回復性
  • 應變性

工程師為了應付一些需求，就會在註解寫上 HACK ，結果五年後還是 HACK。 (其實是feature)

你有聽過測試嗎？

  • 驗證結果符合預期
  • 找出潛在問題
  • 安全防護網
  • 有重構的機會

如何做測試？

你做的是測試嗎？

  • 預期會發生的行為 → 驗證 → 機器
    • 例如：登入
  • 找出意料外的行為 → 測試 → 人
    • 找出預期之外
  • 讓驗證自動化

測試工作量

  • 只要測多出來的程式碼就好嗎？
    • 實際上第四個 Spring 早就得多一倍測試人力

為什麼需要自動化測試

  • 原測試人員的 7 倍

惡性循環

  • 開發時間少
  • 測試問題多
  • 加班趕工上線
  • 發現程式很難改
  • 投入更多資源

加人沒有用，但是大家一樣還在加人 XD

讓測試自動化

有測試就安全了嗎？

  • 覆蓋率
  • 重複
  • 安全性
  • 風格
  • 技術債
    • 應該被償還的

自動化部屬

  • 避免手誤
  • 標準化流程
  • 節省時間
  • 減少人力介入
    • Gitlab 員工案例 😅
    • http://www.ithome.com.tw/news/111634

持續交付流程

  • 兩個人一起開發可以更早發現 BUG (Pair Programing)

Developer

  • 工具不是萬能
  • 人工確認
  • Definition of Done
    • 工具只是輔助人
  • 同步資訊

持續交付流程

  • Pair Programing before CI
  • Code Review

Multiple Stage

  • 確保在各個環境都能執行

環境落差

  • 環境不一致
  • 資料量大
    • QA 環境不見得會有這問題，唯有實際狀況才能顯現。
  • 資料複雜度高
  • 效能問題
  • 架構問題
    • QA 機器沒有錢，都在同一台機器
    • 實際：防火牆等問題

Canary & Rolling Release

  • Production 外加一個 Canary
  • 上新版前先在 Canary 上，流量先從這邊進來測試
  • Rolling update 慢慢導入 Produciton

壓力測試的難題

  • 與現實有落差
  • 人類 vs 機器
    • 機器有限
    • 可能會有測試死角
  • 測試的死角
  • 瞬間的瓶頸
    • 人不會慢慢進來，例如：搶火車票

用壓測軟體測，系統會自己加機器來平衡連線。但實際上並不是這樣，人會一下子全部連進來，機器根本來不及加。

模擬使用者行為

  • PhantomJS
  • Selenium (投影片沒寫？)
  • 根據 Request Log
  • 根據 GA
    • 分佈使用者到底看哪些頁面
  • 客製化腳本
    • 用 Log 去模擬使用者行為寫客製化腳本
  • 同時發動

  • 有效產出

Immutable Infrastructure

網站架設

  • 初期架構
    • WetSite
    • Database
    • Redis

Scalable Application (自動長機器)

  • Stateless
    • 隨時可以複製一台新的，然後上線
  • Recoverable
    • 如果發生問題，原本機器直接砍掉重建一台新的，而且是自動的
  • Reuse
    • 可能是不同國家的
  • Scale out & Scale up
    • 不可能預期這網站是不會垮的

Maintainable

不可能單純靠人工

  • Install
  • Setup
  • Maintain

Infrastruture as Code

  • Terraform
  • CM Tools
    • Ansible
    • Puppet
  • Docker
    • 輕鬆部署 Testing Env

Terraform

  • HCL / JSON
  • 容易閱讀
  • 支援多平台
  • 可匯入現有資源
  • 支援多平台
    • AWS
    • GCP
    • AZure
  • 版本控管
    • 可溯及之前的所有修改
    • 可 review 環境修改

Puppet

  • Provision
  • Package AWS AMI
  • Role Profile Pattern
  • Easy Reuse
  • Auto Scaling Group

Docker

  • 快速建置環境
  • 開發與正式環境一致
  • 環境隔離
  • 資源有效利用
  • 降低維運成本
  • 比虛擬化更輕量
    • 開個 VM 幾分鐘，開個 Docker 幾秒鐘

資源最佳化

  • WebSite 50%
  • Database 20%
  • Redis 40%
